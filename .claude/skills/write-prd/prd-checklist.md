# PRD 编写检查清单

> 完成 PRD 编写后，逐项执行以下检查。任何一项未通过 = PRD 未完成。

---

## 一、结构完整性检查

### 1.1 六章齐全

- [ ] §1 概述包含三小节：定位与目标（1.1）、用户角色（1.2）、前置依赖（1.3）
- [ ] §2 业务流程至少包含主流程 Happy Path（2.1）
- [ ] §3 功能清单覆盖所有功能点，编号连续无跳跃
- [ ] §4 每个功能点至少包含 .1 领域模型 + .2 页面设计 + .3 交互行为 + .5 数据规格
- [ ] §5 全局规则存在且不与 §4 重复（单功能规则放 §4，跨功能规则放 §5）
- [ ] §6 每个功能点至少有一条对应验收标准

### 1.2 元信息头完整

- [ ] 文档标题格式：`# [模块名] 产品需求规格书（PRD）`
- [ ] 元信息包含：模块标识、状态、版本、日期、作者、关联文档路径

## 二、一致性校验（核心质量门禁）

### 2.1 状态机一致性 — 必须逐项核对

对每个包含状态机的功能点：

| # | 校验项 | 通过标准 |
|---|--------|---------|
| C1 | 状态机状态数量 = domain-model 枚举值数量 | 数量一致 |
| C2 | 状态机每个状态名 = domain-model 对应枚举值 | 命名完全一致 |
| C3 | 状态机转换路径覆盖所有合法状态变更 | 无遗漏转换 |
| C4 | §4.X.3 的状态枚举对照表全部标记 ✅ | 无 ❌ |

**操作方法**：在 §4.X.3 中填写状态枚举对照表时，逐行与 `domain-model.md` 中对应实体的 status/state 字段对比。发现不一致立即修正——以 domain-model 为准。

### 2.2 字段一致性

对每个功能点：

| # | 校验项 | 通过标准 |
|---|--------|---------|
| C5 | §4.X.5 输入数据规格中的所有字段 ⊆ §4.X.1 领域模型的关键字段 | 无多余字段 |
| C6 | §4.X.5 输出数据规格中的所有字段 ⊆ §4.X.1 领域模型的关键字段 | 无多余字段 |
| C7 | §4.X.4 校验规则中引用的字段 ⊆ §4.X.5 输入数据字段 | 无幽灵字段 |

**多余字段处理**：如果在数据规格中发现领域模型未定义的字段 → 要么补充到领域模型，要么从数据规格中删除（确认确实不需要）。

### 2.3 功能点完整性

| # | 校验项 | 通过标准 |
|---|--------|---------|
| C8 | §3 功能清单的每一项都在 §4 有对应的 `### 4.X` 章节 | 无遗漏 |
| C9 | §4 的每一项章节编号都在 §3 功能清单中有对应条目 | 无多余 |
| C10 | 功能点编号 F-Mx-NN 从 01 开始连续递增 | 无跳跃 |

### 2.4 验收覆盖率

| # | 校验项 | 通过标准 |
|---|--------|---------|
| C11 | 每个 P0 功能点 ≥ 2 条 AC | 覆盖正常+异常 |
| C12 | 每个 P1 功能点 ≥ 1 条 AC | 至少基本验证 |
| C13 | §6.2 边界异常验收覆盖关键异常场景 | 网络/并发/权限/空数据 |
| C14 | 所有 AC 描述不含模糊词 | 见下方禁用词表 |

**AC 禁用词表**（出现即不通过）：

| 禁用词 | 替换为 |
|--------|--------|
| "功能正常" | 具体操作步骤 + 预期结果 |
| "正确显示" | 具体显示内容/颜色/位置 |
| "可用" | 可执行的操作 + 系统响应 |
| "合理" | 量化指标或具体行为 |
| "适当" | 明确的条件或数值 |
| "良好" | 可测量的 UX 标准 |

## 三、质量门禁

### 3.1 内容完整性

- [ ] 全文无 TBD、TODO、待补充等占位符
  - 如有未确定内容，标记为明确的后续版本（如 `v1.1 待定：XXX`）
- [ ] 全文无歧义表述（已用上述禁用词表扫过）

### 3.2 Mermaid 图表正确性

- [ ] `flowchart TD` 用于业务流程图
- [ ] `stateDiagram-v2` 用于状态机
- [ ] `erDiagram` 用于实体关系图
- [ ] `graph TD` 用于布局结构图
- [ ] 节点语法符合约定（用户操作=`[]`, 系统=`()`, 判断=`{}`, 数据=`[[]]`, API=`>]`）
- [ ] 泳道使用 `subgraph` 且命名含 Emoji + 执行者名称

### 3.3 引用有效性

- [ ] 关联文档路径格式正确（相对路径 + 锚点）
- [ ] 标注了尚未创建的文档及其预期位置

### 3.4 AI 编码提示质量

对每条 AI 编码提示（§4.X.6）：

- [ ] 记录的是**实现陷阱或注意点**，不是技术方案
- [ ] 包含**原因说明**（为什么要注意这点）
- [ ] 不与 §4 其他子节重复信息
- [ ] 对编码 Agent 有实际指导价值（非泛泛而谈）

## 四、编写顺序自检

确认你遵循了 **Skeleton-First** 顺序：

- [ ] 第 1 步：写了 §1 概述（知道做什么）
- [ ] 第 2 步：画了 §2 Happy Path 流程图（知道怎么做）
- [ ] 第 3 步：列了 §3 功能清单（知道有哪些功能点）
- [ ] 第 4 步：逐个填充 §4 功能点详细设计（血肉）
- [ ] 第 5 步：提取 §5 跨功能全局规则（收口）
- [ ] 第 6 步：写 §6 验收标准（可测试性保障）

如果发现是从 §4 开始线性填写的 → 建议重排，先补骨架再确认完整性。
