# Write PRD

## Overview

结构化产品需求文档编写 Skill。核心原则：**先骨架后血肉**——先定业务流程和功能清单（§2+§3），再逐功能点填充详细设计（§4）。输出物必须同时满足人可读和 AI 可消费两个目标。

## When to Use

```
需要输出 PRD？
  ├── 是 → 产品原型阶段 / SDLC Step 2 → ✅ 使用本 Skill
  ├── 是 → 需求已明确但缺少结构化规格 → ✅ 使用本 Skill
  └── 否 → 不使用
```

**不适用场景**：
- 技术方案/架构设计（归 Step 3）
- 一句话需求描述（不需要完整 PRD）
- 纯 UI 原型（无行为逻辑）

## Core Pattern: Skeleton-First Writing

| 阶段 | 输出章节 | 关键动作 | 禁止事项 |
|------|---------|---------|---------|
| **1. 定位** | §1 概述 | 写清模块定位、用户角色、前置依赖 | 不要跳过，即使看起来简单 |
| **2. 骨架** | §2 业务流程 + §3 功能清单 | 先画 Happy Path 流程图 → 拆功能点 → 编号 F-Mx-NN | 不要从 §4 开始填 |
| **3. 血肉** | §4 功能点详细设计 | 逐个功能点按 6 子节模板填充 | 不要遗漏 .3 交互行为、.4 业务规则或 .5 数据规格 |
| **4. 收口** | §5 跨功能规则 + §6 验收标准 | 提取全局规则、写可测试的 AC | AC 不能是"功能正常" |

## Quick Reference

### 文档结构（6 章）

| 章 | 内容 | 必填 | 产出量级 |
|----|------|:----:|---------|
| §1 概述 | 定位 / 角色 / 依赖 | ✅ | ~30 行 |
| §2 业务流程 | Happy Path + 异常分支 + 页面交互 | ✅ 主流程必填 | ~60 行/Mermaid |
| §3 功能范围 | 清单(编号) / Out of Scope / 术语 | ✅ | ~20 行 |
| §4 功能点详细设计 | 每功能点 6 子节（见模板） | ✅ 核心 | ~80 行/功能点 |
| §5 跨功能规则 | 全局状态机 / 校验 / 交互约定 / 权限 | 按需 | ~40 行 |
| §6 验收标准 | 功能 AC / 边界异常 AC / UI/UX AC | ✅ | ~30 行 |

### 编号体系

| 前缀 | 格式 | 示例 |
|------|------|------|
| `F-` | 功能点 | `F-M1-03` |
| `B-` | 业务规则（功能点内） | `B-M1-05` |
| `G-` | 全局规则（跨功能点） | `G-M1-02` |
| `AC-` | 验收标准 | `AC-M1-12` |

### Mermaid 节点约定

| 含义 | 语法 | 示例 |
|------|------|------|
| 用户操作 | `[文本]` | `[点击新建按钮]` |
| 系统/页面 | `([文本])` | `(显示创建表单)` |
| 判断 | `{文本}` | `{校验通过？}` |
| 数据/状态 | `[[文本]]` | `[[status=draft]]` |
| API | `>文本]` | `>POST /api/tasks]` |

## §4.X.4 业务规则：CRUD 分操作展开（强制）

**本规则优先级高于模板中的默认格式。** 每个功能点的业务约束（`§4.X.4`）**必须按 CRUD 操作维度逐条展开**，不允许省略任何操作、不允许用概括性描述一笔带过。

### 要求

对每个功能点，按其涉及的操作（不限于 CRUD，也可能是审批/导入/导出等），**逐一描述**：

| 操作 | 必须写清的内容 |
|------|--------------|
| **查询（Read/List）** | 查询条件、排序规则、分页策略、权限过滤、数据范围（全量/本人/部门）、缓存策略 |
| **新增（Create）** | 前置校验（哪些字段必填/唯一/格式）、业务校验逻辑（如重名检测、配额检查）、默认值规则、关联数据自动创建 |
| **修改（Update）** | 可编辑字段清单（哪些能改哪些不能改）、乐观锁/版本控制策略、修改范围限制（只能改自己的？）、级联更新规则 |
| **删除（Delete）** | 物理删除 vs 逻辑删除、前置条件（有关联数据时怎么处理）、批量删除规则、恢复机制（如逻辑删除） |
| **其他操作** | 审批流、状态变更、导入导出等，同样需展开处理逻辑和注意点 |

### 禁止

| 禁止行为 | 正确做法 |
|---------|---------|
| "支持增删改查" 一句话带过 | 每个操作单独一段，写清业务逻辑 |
| 只写 Happy Path | 必须包含异常分支和边界条件 |
| 跳过"看起来简单"的操作 | 即使简单的查询也要写出过滤和排序规则 |
| 把规则混在交互行为（.3）或数据规格（.5）中 | 业务规则统一收在 .4 中，.3 只写交互流程，.5 只写字段定义 |

### 编号

每个操作下的具体规则使用 `B-Mx-[序号]` 编号，全局递增。

## 交互约束：逐节确认（强制）

**本约束优先级高于所有其他流程规则。** PRD 编写过程中，AI 必须在每节前后与用户交互确认。

### 确认流程

```
对每一章（§1 ~ §6）：

  ┌─ 写前确认 ─────────────────────────────┐
  │ 1. 输出该节的【写作思路】（3-5 条要点）   │
  │    - 覆盖哪些内容、重点是什么             │
  │    - 与前序章节的衔接关系                 │
  │    - 预判可能的决策点或需要用户输入的地方     │
  │ 2. 使用 AskUserQuestion 询问：            │
  │    "以上是 §X 的写作思路，是否可以开始写？"  │
  │    选项：[开始写] / [调整思路] / [跳过本节] │
  └──────────────────────────────────────────┘
                    ↓ 用户确认后
  ┌─ 执行编写 ──────────────────────────────┐
  │ 按照确认后的思路编写该节完整内容           │
  └──────────────────────────────────────────┘
                    ↓ 编写完成后
  ┌─ 写后确认 ──────────────────────────────┐
  │ 1. 展示该节的【完成摘要】（关键产出物）     │
  │ 2. 使用 AskUserQuestion 询问：            │
  │    "§X 已完成，是否满意？可以继续下一节？"   │
  │    选项：[继续下一节] / [修改本节] / [暂停] │
  └──────────────────────────────────────────┘
```

### 规则细节

| 规则 | 说明 |
|------|------|
| **每章必确认** | §1 到 §6，无一例外。即使看起来简单的章节（如 §1 概述）也必须走完整确认流程 |
| **写前输出写作思路** | 不是简单说"我要写 §X"，而是列出具体的写作要点、重点、预判决策点 |
| **写后展示完成摘要** | 让用户快速判断质量，而不是要求用户重读整节内容 |
| **用户可随时调整** | "调整思路"时根据反馈重新规划；"修改本节"时根据具体意见修订 |
| **允许跳过/暂停** | 用户选择"跳过本节"时标记为 SKIP 并记录原因；"暂停"时保存当前进度 |
| **不堆积多章** | 严禁一次性写完多章再统一确认——这违背了逐节确认的本意 |

### 写前确认模板

在 AskUserQuestion 中使用以下格式呈现写作思路：

```
## §{N} {章节名} — 写作思路

**覆盖内容：**
- 要点 1
- 要点 2
- ...

**与前序章节的衔接：**
- 基于 §{N-1} 的 {具体产出}，本章将 ...

**预判决策点：**
- 决策点 1（可能需要你确认）
- ...
```

### 写后确认模板

```
## §{N} {章节名} — 完成摘要

**核心产出：**
- {关键交付物 1}
- {关键交付物 2}

**数据点：** {行数 / 功能点数 / AC 数量 等}

**待下一节依赖：** {如果有的话}
```

## Implementation

### 完整模板 → [prd-template.md](./prd-template.md)

6 章完整模板，含每个子节的填写示例和格式规范。编写时以此为准。

### 检查清单 → [prd-checklist.md](./prd-checklist.md)

完成后逐项检查：结构完整性、一致性校验（状态机/字段/功能点/验收覆盖）、质量门禁。

## Common Mistakes

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 从 §4 开始线性填充 | 功能点遗漏、编号跳跃 | 先 §2+§3 定骨架，确认功能点全了再填 §4 |
| 验收标准写"功能正常" | 无法测试、无法自动化 | 用 Given-When-Then 格式写可执行步骤 |
| 状态机与枚举值不一致 | 编码阶段发现矛盾 | 写完状态机立即做对照表校验 |
| AI 编码提示写成技术方案 | 与 Step 3 重复、膨胀 | 只记录实现陷阱和注意点 |
| 跳过 Out of Scope | 范围蔓延、评审时被挑战 | 明确写出"不做什么"及原因 |
| §5 与 §4 规则重复 | 维护时改了一处漏另一处 | 全局规则只放 §5，单功能规则放 §4.X.4 |
| 业务规则一笔带过（"支持增删改查"）| 编码阶段遗漏边界逻辑 | 按 CRUD 分操作逐条展开，每条写清处理逻辑和注意点 |

## Red Flags — STOP and Check

出现以下信号时停下来检查：

- §3 功能清单中有某项在 §4 找不到对应设计 → **补齐 §4 或从 §3 移除**
- §6 验收标准中存在"正常"、"正确"、"可用"等模糊词 → **重写为可执行步骤**
- 状态机的状态列表与数据规格枚举值数量不同 → **做对照表逐项核对**
- §4.X.5 数据规格中的字段不在 §4.X.1 领域模型中 → **补充领域模型或删除多余字段**
- §4.X.4 业务规则中出现概括性描述（如"支持增删改查"）→ **按 CRUD 分操作逐条展开**
- 文档中出现 TBD / TODO / 待补充 → **要么现在填上，要么标记为明确的后续版本**

**以上任何一项未通过 = PRD 未完成。**
