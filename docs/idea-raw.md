# 关于Harness Creator的核心想法

## 1、Harness Creator 在你心中的真实定位是什么？

我认为是一套工具，它的输入可以是已有的工程规范，以及基于不同的工程规范产生的各种输出物（比如产品设计文档、交互原型），它的核心是对软件工程不同环节的方法论的高度整合和抽象，它的输出是针对不同的 coding/UX设计/测试执行等 agent 的、可持续监控和迭代（因为 agent 和模型都在变化）的一套可用落地工具（skills、plugins 等，具体要看不同 agent 的工作模式）

## 2、输入→输出的过程比预想的更复杂

### 2.1 输入的复杂性

输入不只是一份规范的简单转换。同样是「产品文档」这类输入，可能有**几十种不同风格**的源文档。这意味着项目不能假设输入是标准化、结构化的。

### 2.2 中间处理过程：从混乱输入到固化 Skill

从原始输入到最终产出的 Skill，中间需要一个**分析→研究→确认→固化**的过程：

- **分析**：AI 解读输入文档，提取方法论要素（流程、规则、格式要求、评判标准）
- **网络搜索**：补充上下文（如某个术语的行业通用定义、最佳实践）
- **问答确认**：与用户（项目拥有者/领域专家）交互，澄清模糊之处、确认关键决策
- **固化**：将经过验证的理解编译为结构化的 Skill 定义

### 2.3 多版本管理

同一个工程环节的方法论，可能产出**多个版本的 Skill**：
- 不同输入源文档风格 → 不同版本
- 不同严格程度 → 不同版本（如 strict / relaxed 模式）
- 不同团队偏好 → 不同版本
- 需要支持版本的保存、对比、切换

## 3、与现有 Agent 框架的关系：跨框架的方法论中间件

Harness Creator 不绑定任何特定 Agent 框架（Claude Code / Codex / Cursor / Cline / MetaGPT / OpenHands ...）。它的定位是：

> **定义一套描述语言（类似 MDL — Methodology Description Language），用来表达软件工程不同环节的要求、规范、格式；然后通过编译器针对不同 Agent 做针对性的编译输出**

这意味着：
- Harness Creator 的核心产出是 **MDL 描述的方法论定义**
- 编译器负责将 MDL → 各 Agent 框架的原生格式（Skills / Plugins / Extensions / Config 等）
- 新增一个 Agent 框架的支持 = 新增一个编译器后端（而非重写方法论本身）

## 4、切入点和优先级

### 4.1 MVP 切入点：Coding Agent

优先面向编程类 Agent（Claude Code / Codex / Cline 等）产出工具。原因：
- 编程 Agent 生态最成熟，工具形态相对清晰（Skills、MCP、Extensions 等形态已有共识）
- 工程规范体系最完善（编码规范、Code Review 标准、Git 工作流等）
- 可快速验证「方法论 → Agent 工具」的完整链路

### 4.2 后续扩展方向（非 MVP）

- UX 设计 Agent（Figma AI 插件、LLM 驱动的原型生成）
- 测试 Agent（SWE-agent、OpenHands 的代码执行能力）
- 更多 SDLC 环节...

## 5、「持续监控和迭代」的初步判断

这是一个重要但复杂的能力：
- Agent 和模型在快速变化 → 同一个 Skill 在新版模型上效果可能变差
- 工程方法论本身也在演进 → Skill 需要跟进更新

**当前判断**：记录到远期待办中，MVP 阶段暂不考虑。但架构上需要留出扩展空间。
