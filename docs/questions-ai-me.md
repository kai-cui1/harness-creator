## Round 1 — 2026-04-22（基于架构设计 v0.1.0 的对齐讨论）

### 背景

AI 完成了 Harness 架构设计文档 v0.1.0（五层模型：Presentation → Task Definition → Capability Abstraction → Framework Adapter → LLM Provider），但用户认为存在方向性偏差，需要深入对齐。

---

### Q1: Harness Creator 的真实定位是什么？

**AI 初始理解（v0.1.0 偏差版）**：
Harness Creator = 编程 Agent 的多框架适配层，核心是让同一套 Task Definition 在 Claude Code / Codex / Cursor 上都能跑。类似 CCR 的上层抽象。

**用户回答**：
> 我认为是一套工具，它的输入可以是已有的工程规范，以及基于不同的工程规范产生的各种输出物（比如产品设计文档、交互原型），它的核心是对软件工程不同环节的方法论的高度整合和抽象，它的输出是针对不同的 coding/UX设计/测试执行等 agent 的、可持续监控和迭代（因为 agent 和模型都在变化）的一套可用落地工具（skills、plugins 等，具体要看不同 agent 的工作模式）

**关键纠正**：不是「任务编排器/适配层」，而是「方法论→Agent 工具」的转化平台。

---

### Q2: 输入→输出的过程有多复杂？输入是什么形态？

**AI 猜测**：输入是标准化的工程规范文档，一次性转换输出 Skill。

**用户回答**：
> 整个过程比预想的更复杂：
> 1. **输入非常多样**：同样是「产品文档」，可能有几十种不同风格的源文档。不能假设输入是标准化、结构化的。
> 2. **中间有处理过程**：从原始输入到最终 Skill，需要「分析 → 网络搜索 → 问答确认 → 固化」的过程。
> 3. **多版本管理**：同一个环节的方法论可能产出多个版本的 Skill（不同风格、不同严格程度、不同团队偏好），需要保存、对比、切换。

**核心洞察**：这不是 ETL（Extract-Transform-Load），而是一个 **交互式的知识工程过程**。

---

### Q3: Harness Creator 与现有 Agent 框架的关系？

**AI 给出 4 个选项**：
1. Harness 是 Agent 框架的「上游」（产出 Skills 给框架用）
2. Harness 自身就是一套 Agent 框架
3. Harness 是跨框架的「方法论中间件」（定义 MDL 描述语言 + 编译器）
4. 以上都不是

**用户回答**：**选项 3 — 跨框架的方法论中间件**

> 定义一套描述语言（类似 MDL — Methodology Description Language），用来表达软件工程不同环节的要求、规范、格式；然后通过编译器针对不同 Agent 做针对性的编译输出。

**含义**：
- 核心产出 = MDL 描述的方法论定义
- 编译器负责 MDL → 各框架原生格式
- 新增框架支持 = 新增编译器后端（不重写方法论）

---

### Q4: 「持续监控和迭代」怎么落地？

**AI 追问**：是指 Skill 在新模型上效果变差的自动检测？还是方法论演进的跟进？还是两者？

**用户回答**：
> 这是个好问题，目前我觉得考虑进去会比较复杂，我倾向于记录在我们的远期待办里，但是 MVP 不需要考虑。

**结论**：MVP 暂不考虑，但架构留扩展空间。

---

### Q5: 从哪类 Agent 切入 MVP？

**AI 分析了选项**：Coding / UX Design / Testing，给出了各自成熟度分析。

**用户回答**：**从 Coding Agent 切入**

理由（AI 补充推断）：编程 Agent 生态最成熟，工具形态清晰（Skills/MCP/Extensions 已有共识），工程规范体系最完善。

## Round 1 总结：AI 对项目定位的重构理解

```
修正前（v0.1.0 偏差）          修正后（基于 idea-raw.md）

任务编排器                     方法论编译器
多框架适配层                    跨框架中间件
"让 Task 到处能跑"              "让方法论变成工具"
输入: 结构化任务定义              输入: 杂乱的多种风格规范文档
输出: 执行结果                   输出: 各框架可用的 Skills/Plugins
一次性转换                      交互式知识工程过程（分析→搜索→确认→固化）
单一版本                        多版本管理
```

**一句话重新定义**：
> Harness Creator 解决的核心问题是 —— **如何将一堆杂乱的、风格各异的工程规范文档，通过 AI 驱动的交互式过程，转化为各个 Agent 框架可以直接使用的高质量、可持续迭代的方法论工具（Skills/Plugins）。**

## Round 3 — 2026-04-23（MDL 形态定义 + 首个 E2E 验证）

### 完成事项

#### 1. 行业交互设计规范标准调研（待办 2）→ 已完成

**产出**：`docs/industry-design-standards-analysis.md`

**核心发现**：
- 三大系统（M3 / Ant Design / HIG）提取出 **16 类通用方法论要素**，分 3 个 Tier
- Tier 1（必须）：设计原则、组件规格、视觉基础、交互流程、异常处理、信息架构
- 翻译机 3.0 与行业标准高度同构 — 「原则→组件→交互→异常」链条普遍存在
- MDL 建议采用 **4 层表达模型**：元层 → 原则层 → 规范层 → 实例层

#### 2. MDL 形态定义（问题 B）→ 已完成

**产出**：`docs/mdl-form-proposal.md`（v0.1 定稿）

**5 项设计决策全部敲定**：

| 决策 | 结论 |
|------|------|
| 表达形态 | **混合形态**：YAML Frontmatter + Markdown 主体 |
| 文件扩展名 | **`.mdl.md`** |
| Schema 校验 | MVP 需要，轻量级（只校验 L1+L3 骨架） |
| 多文件 vs 单文件 | 支持两种模式，MVP 先实现单文件 |
| 自然语言结构化程度 | **三级梯度**：YAML(强) → 表格(半) → 段落(弱) |
| 格式兼容性 | 遵循 GFM + YAML Front Matter 标准 |

#### 3. E2E 流程验证（问题 A）→ 已完成

**验证对象**：磁贴样式规范（翻译机 3.0）
**产出**：`harnesses/tile-component-spec.mdl.md`（首个真实 MDL 文件）

**四步流程实际执行记录**：

| 步骤 | 做了什么 | 耗时/复杂度 | 人参与度 |
|------|---------|------------|---------|
| **Analyze** | 读取 Axure HTML 源文件，提取文本规则（5 条总则）、结构化表格（9×6）、设计理由（3 个故事）、隐式知识推断（布局/像素值/CSS 层次） | 中等 — 文本部分可自动解析，图片依赖推断 | 低（AI 可独立完成大部分） |
| **Search** | 搜索 UI Card/Tile 设计最佳实践，补充行业上下文（7 种模式、8 条实践） | 低 — 补充性搜索 | 无需人参与 |
| **Confirm** | 向用户呈现 3 类确认项（规则完整性、隐式推断合理性、MDL 结构合适性），用户全部确认 | 低 — 3 个多选项确认 | 中（用户审核确认） |
| **Solidify** | 输出完整 MDL 文件（L1-L4 四层完整，含审查检查清单） | 低 — 格式化输出 | 无需人参与 |

**关键发现**：
- Analyze 步骤中，**可自动提取 ~70%** 的内容（文本规则 + 表格 + CSS 结构）
- **~30% 需要确认**（隐式推断、设计理由解读、待定项判断）
- Confirm 步骤是**不可省略的** — 用户对文档的理解是 AI 无法替代的
- 最终 MDL 文件质量高度依赖于 Confirm 环节的质量

## Round 5 — 2026-04-27（Model Integration Layer 调研）

### Q: 传统软件系统 + LLM/Agent 混合架构的 Model 集成需要做哪些工作？

**用户原话**：
> 如果我希望自己开发一个系统，这个系统保留了传统软件系统的能力，同时在系统里又可以集成大模型，本质上就是一个软件系统+Agent的一个架构（但是又不是纯粹作为一个agent存在），那么这个系统在集成model部分需要做哪些工作？

**调研产出**：`docs/model-integration-research.md`

### 核心结论

MIL（Model Integration Layer）是传统软件与 LLM 能力之间的**工程契约层**，包含 **6 大子系统**：

| # | 子系统 | 核心职责 | MVP 优先级 |
|---|--------|---------|-----------|
| 1 | **调用网关 (Gateway)** | 统一 API、多 Provider 适配、路由、故障转移 | P0 |
| 2 | **Prompt 管理** | 模板引擎、版本管理、变量 Schema、链式编排 | P0 |
| 3 | **上下文组装 (Context Assembler)** | Token 预算管理、优先级排序、RAG 集成 | P0 |
| 4 | **响应处理 (Response Handler)** | 结构化输出解析、校验、重试、流式 | P0 |
| 5 | **运行时保障 (Runtime)** | 重试/超时/限流/缓存/降级/异步 | P0 |
| 6 | **可观测性 (Observability)** | 日志/追踪/指标/成本/安全审计 | P0-P1 |

### 对 Harness Creator 的直接意义

1. **Harness Creator 自身就是混合架构**：传统部分（MDL 解析/渲染/校验/Pipeline）+ LLM 增强部分（Analyze/Search/推断）
2. **去掉 LLM 系统仍可运行**：核心价值不依赖 LLM，LLM 是效率放大器
3. **MIL 是 Phase 3 的基础设施**：在实现 Phase 3（LLM 集成）之前，需要先搭建 MIL
4. **MVP 建议**：不引入 LangChain，自行实现轻量 MIL（调用点明确且有限，不需要复杂编排）

### 实施路线图建议

- **Phase 0**（1-2天）：基础连通 — 单 Provider 适配 + 最简调用
- **Phase 1**（3-5天）：核心 MIL — 统一模型 + Prompt 模板 + 结构化输出 + 重试日志
- **Phase 2**（3-5天）：可靠性与效率 — Token 管理 + 上下文组装 + 缓存 + 成本追踪
- **Phase 3**（后续）：高级能力 — RAG / Prompt A/B / 降级策略 / 流式 / 质量评估

### 待继续探讨的问题

- MIL 的技术选型最终决策（是否用 LiteLLM 作为 Gateway 基础？）
- Phase 3 的具体切入方式（先做哪个 LLM 调用点？）
- Token 成本预算的初始策略（免费额度 vs 付费 API vs 本地模型）
- 是否需要 RAG 能力（行业知识库检索）作为 MVP 的一部分
