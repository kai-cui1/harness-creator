### 1.3 内容载体形态分析

这份文档的内容以**三种形态**共存：

| 形态 | 占比估计 | 特点 | 可提取性 |
|------|---------|------|---------|
| **文本规则**（HTML 中 `<span>` 文本） | ~20% | 显式写明的规范条款、设计理由、数值约束 | 高 — 可直接解析 |
| **视觉原型**（`<img>` 引用 PNG/SVG） | ~70% | 实际界面截图/线框图，包含布局、间距、色彩 | 中 — 需要视觉理解能力 |
| **结构化数据**（Axure `<table>` / `data.js`） | ~10% | 表格（如磁贴规格表）、组件属性、页面连线关系 | 高 — 可程序化提取 |

**关键发现**：这是一份典型的**「图文混排 + 以图为主」**的交互设计文档。纯文本解析只能获取 ~30% 的方法论信息，剩余 ~70% 依赖对视觉原型的理解。

## 二、方法论要素提取

### 2.1 设计原则层（Design Principles）

从 `项目背景.html` 提取到 **三大设计策略**，每个策略有明确的用户行为假设：

| 原则 | 定义 | 行为假设 |
|------|------|---------|
| **易学** | 遵循用户在手机端已形成的习惯，将无意识行为转化为可见之物 | 用户已有手机操作心智模型 |
| **高效（便捷）** | 简化界面和操作流程，扩大展示区域及操作热区，利用手势操作 | 屏幕小（3.1 寸），误触成本高 |
| **愉悦** | 在基本功能满足的情况下增加更多用户关怀 | 工具类产品需要情感化设计 |

**MDL 表达需求**：需要能表达「设计原则 → 行为假设 → 具体规则」的推导链。

### 2.2 全局组件规范层（Global Component Specs）

从 `hud、toast、弹窗、空白页、时间等规范.html` 和 `磁贴样式规范（new）.html` 提取：

#### 2.2.1 HUD（Head-Up Display）规范

- **Loading HUD**：全屏半透明蒙层 + 居中 loading 动画
- **时间显示格式**：「2月9日 星期二」+ 「15:16」（日期+时间分离）
- **状态栏固定元素**：左侧返回箭头 + 标题 + 右侧图标组

#### 2.2.2 Toast / 弹窗规范

| 类型 | 触发条件 | 展示内容 | 操作选项 | 自动消失 |
|------|---------|---------|---------|---------|
| **Toast 提示** | 操作反馈 | 短文本（如「短信已发送」） | 无 | 5 秒 |
| **确认弹窗** | 需用户决策 | 标题 + 描述文本 | 主操作 + 取消 | 否 |
| **异常弹窗** | 错误/断网/超时 | 图标 + 标题 + 描述 | 重试/连接网络等 | 否 |

#### 2.2.3 磁贴（Tile）设计系统

这是文档中**最完整的组件规范案例**，包含：

```
磁贴规范 = {
  尺寸变体: ["wide", "small"],           // 两种尺寸
  固定信息: ["应用icon", "应用名称"],       // 必填
  动态信息: ["语种", "流量套餐"],           // 选填（本期不实现）
  背景色: "固定纯色（由UI给出色号）",
  约束: {
    不支持用户自定义尺寸: true,
    不支持用户自定义位置顺序: true,
    不支持卸载: true,
    不承载通知/角标类运营信息: true,
    后台可配置上下架: true
  },
  各功能磁贴规格表: [                       // 10 个功能的完整矩阵
    {功能, 尺寸, 背景, APP名称, APP logo, 动态信息}
  ],
  设计理由: [                              // 每个元素都有"故事"
    {背景: "用户通过色块大小/颜色辅助记忆功能位置"},
    {logo: "用户通过logo形状识别应用功能"},
    {名称: "用户通过文字识别应用功能"}
  ]
}
```

**这是一个完美的 MDL 表达候选** — 它同时包含了：
- 结构化约束（schema）
- 枚举值（具体功能列表）
- 设计理由（rationale）
- 排除性规则（不做什么）

### 2.3 交互模式层（Interaction Patterns）

#### 2.3.1 硬件按键映射

从 `硬件与界面的使用流程.html` 提取 SOS 流程：

```
SOS 交互流程 = [
  {步骤: 1, 条件: "长按背部SOS键 ≥ 3秒", 反馈: "设备震动 2秒"},
  {步骤: 2, 判断: "网络是否 connected?"},
  {步骤: 2a, 分支: "未联网", 弹窗: "网络未连接，SOS暂不能用", 按钮: ["连接网络", "取消"]},
  {步骤: 2b, 分支: "已联网", 判断: "SOS 功能是否开启?"},
  {步骤: 3a, 分支: "未开启", 弹窗: "紧急联络功能尚未开启", 按钮: ["去开启", "取消"]},
  {步骤: 3b, 分支: "已开启", 动作: "发送短信+位置给紧急联络人"},
  {步骤: 4, 反馈: "Toast '正在发送短信' (5秒) → Toast '短信已发送'"}
]
```

**关键特征**：
- 硬件触发 → 软件响应的跨域映射
- 多层条件分支（嵌套 if-else）
- 每步都有明确的感官反馈（震动/弹窗/Toast）
- 时序约束（长按 ≥ 3s，Toast 持续 5s）

#### 2.3.2 手势交互

从 `语音翻译流程（长按）.html` 和 `长按.html` 提取：

| 手势 | 阈值 | 响应 |
|------|------|------|
| 短按 | < 500ms | 切换语种/触发简单动作 |
| 长按 | ≥ 500ms | 进入录音/评测模式 |
| 超长按 | ≥ 3s | 触发 SOS 等特殊功能 |

### 2.4 异常处理模式层（Exception Handling）

从 `异常情况.html` 提取到一套**统一的异常处理框架**：

```
异常处理模式 = {

    错误描述: "人性化文案（非技术术语）",
    恢复操作: "引导性按钮文案",
    返回路径: "左上角返回按钮（始终存在）"
  },
  特殊标注: "口语评测建议做成本地资源（红色文字标注）"
}
```

## 三、显式规则 vs 隐式知识图谱

### 3.1 显式规则（可直接提取）

| 类别 | 示例 | 来源页面 |
|------|------|---------|
| 数值约束 | 屏幕 480×800px，长按 ≥ 3s，Toast 5s，加载超时 8s | 项目背景 / 异常情况 |
| 枚举值 | 磁贴只有 wide/small 两种 | 磁贴样式规范 |
| 文案模板 | "服务不稳定，请稍后再试"、"请先连接网络，再使用此功能" | 异常情况 |
| 排除性规则 | 磁贴不支持自定义尺寸/位置/卸载/通知 | 磁贴样式规范 |
| 条件分支逻辑 | SOS 的 if-else 流程树 | 硬件与界面流程 |
| 变更追踪 | 修订记录表（序号/时间/需求名/修订内容/提出人） | 第一版交互修订记录 |

### 3.2 隐式知识（依赖推理/视觉理解）

| 类别 | 示例 | 为什么是隐式的 |
|------|------|--------------|
| **布局约定** | 状态栏高度 20px（从 data-left/top 推断） | 嵌入在 Axure 坐标数据中，非直接声明 |
| **视觉层次** | 一级标题 > 二级标题 > 正文 > 辅助文字的字号/字重/颜色阶梯 | 通过 CSS class 名称（_一级标题/_二级标题/_文本段落）间接表达 |
| **一致性惯例** | 所有异常页面共用同一套 header 结构（返回+标题+图标） | 跨页面的重复模式，无单独声明 |
| **设计决策理由** | "本期不实现动态信息" — 为什么？ | 只有结论，缺少决策背景 |
| **跨页面引用** | "本期可先按照翻译机2.0的逻辑来做" | 引用外部文档（2.0 版本规范） |
| **待定项语义** | 多处标记"（待定）"或"（可能会去掉）" | 状态标记，需判断是否纳入 Skill |

### 3.3 对 Harness Creator 的启示

```
隐式知识占比估算：~60-70%

这意味着：
├── 纯文本解析（ETL 思路）行不通
├── 必须具备视觉理解能力（或人工辅助标注）
├── 「问答确认」环节不可或缺 — 很多隐式知识需要设计师口头补充
└── 「固化」产物必须同时保留规则 + 原始依据（可追溯）
```

## 四、作为 MVP 验证输入的适配度评估

### 4.1 优势

| 维度 | 评分 | 说明 |
|------|------|------|
| **方法论完整性** | ★★★★★ | 涵盖原则→组件→交互→异常→硬件映射，链条完整 |
| **真实复杂度** | ★★★★★ | 126 页、多层级、图文混排、有变更记录 — 不是 toy example |
| **规则密度** | ★★★★☆ | 全局规范页和异常处理页的规则非常密集 |
| **领域知识可验证** | ★★★★★ | 用户（项目拥有者）就是该文档的设计者，能准确判断产出质量 |
| **输出形态清晰** | ★★★★☆ | 可转化为：审查 Skill（检查新原型是否符合规范）/ 生成 Skill（基于规范生成新页面） |

### 4.2 挑战

| 维度 | 风险 | 说明 |
|------|------|------|
| **图片依赖** | 高 | ~70% 内容在 PNG 图片中，需要视觉理解或 OCR+版面分析 |
| **外部依赖** | 中 | 引用翻译机 2.0 规范、UI 色号（由 UI 给出） |
| **碎片化** | 中 | 126 个独立页面，缺少全局索引/交叉引用表 |
| **硬件特殊性** | 低 | SOS 按键等硬件交互是翻译机特有的，但方法论可泛化 |

### 4.3 建议的首个端到端例子

**推荐**：从「磁贴样式规范」这一个页面开始做完整的 analyze→search→confirm→solidify 流程。

理由：
1. 自包含（不依赖外部文档）
2. 规则完整（有设计理由、尺寸、颜色、排除项、枚举表）
3. 可验证（给定一个新功能，能否正确生成其磁贴规格？）
4. 复杂度适中（不是最简单的，也不是最复杂的）

## 五、对 MDL 设计的初步启发

基于这份真实文档的分析，MDL 至少需要能表达以下维度：

### 5.1 必须表达的维度（Tier 1 — 本文档已证实必要）

| MDL 维度 | 对应本文档的证据 |
|----------|----------------|
| **设计原则 (Principles)** | 易学/高效/愉悦 三大策略 |
| **组件规格 (Component Spec)** | 磁贴的 schema + 约束 + 枚举 |
| **交互流程 (Interaction Flow)** | SOS 的条件分支树、语音翻译的长按流程 |
| **异常模式 (Exception Pattern)** | 统一的异常页面结构（图标+文案+操作） |
| **信息架构 (IA)** | 页面层级树（Home → 功能页 → 设置页） |
| **设计理由 (Rationale)** | 每个 UI 元素的"故事" |
| **排除性约束 (Negative Constraints)** | "不支持"/"不承载"/"本期不做" |

### 5.2 可能需要的维度（Tier 2 — 本文档暗示但未充分展开）

| MDL 维度 | 推测依据 |
|----------|---------|
| **版本管理 (Versioning)** | 修订记录表的存在说明需要版本维度 |
| **状态机 (State Machine)** | 页面切换手势暗示了状态转换 |
| **时序约束 (Temporal Constraints)** | 长按 3s、Toast 5s、超时 8s |
| **硬件抽象 (Hardware Abstraction)** | SOS 按键→软件动作的映射 |
| **国际化 (i18n)** | 多语种翻译场景天然需要 |

### 5.3 MDL 的目标读者

基于本分析，MDL 应该是**双读者**设计：

```
┌─────────────────────────────────────────┐
│            MDL 双读者模型               │
├────────────────────┬────────────────────┤
│   人读（设计师/PM）  │   机读（编译器）    │
│                    │                    │
│ • 自然语言描述      │ • 结构化 Schema     │
│ • 设计理由/故事     │ • 约束表达式        │
│ • 决策背景          │ • 枚举值/默认值      │
│ • 待定项标记        │ • 条件逻辑（if-else）│
└────────────────────┴────────────────────┘
```
