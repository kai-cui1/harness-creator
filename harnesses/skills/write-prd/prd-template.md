**Mermaid 节点形状约定**：

| 含义 | 语法 | 示例 |
|------|------|------|
| 用户操作 | `[文本]` | `[点击新建按钮]` |
| 系统/页面行为 | `([文本])` | `(显示创建表单)` |
| 判断/条件 | `{文本}` | `{校验通过？}` |
| 数据/状态 | `[[文本]]` | `[[status=draft]]` |
| API 调用 | `>文本]` | `>POST /api/projects]` |
| 开始/结束 | `([*])` | — |

**文本转义规则（强制）**：

Mermaid 解析器会将以下字符误认为语法标记，节点文本包含时**必须用英文双引号 `"..."` 包裹**：

| 危险字符 | 会被误认为 | 正确写法 |
|---------|-----------|---------|
| `(` `)` `[]` `{}` | 节点形状 / 判断 / 连接括号 | `"确定归档（XXX）"` ✅ |
| `/` `\` | 备选路径分隔符 | `"路径: src/utils"` ✅ |
| 中文标点 `「」` `！？` 《》 | Unicode 未被支持 | `"确定归档 XXX？" ` ✅ |
| `#` `*` `.` `@` | 特殊语法字符 | `"v1.0.0 发布"` ✅ |

**核心规则：文本含任何非字母数字+空格+中文汉字的字符时，统一用双引号包裹。宁可多加引号也不要冒渲染失败的风险。**

**示例对比**：

```mermaid
## ❌ 错误
U6[确认归档弹窗] %% 归档流程复用此节点概念
M2(弹窗内容："确定归档「XXX」？...")

## ✅ 正确
U6["确认归档弹窗 %% 归档流程复用此节点概念"]
M2("弹窗内容：确定归档 XXX？归档后该项目将不可编辑")
```

**泳道约定**：
- 泳道代表活动执行者（用户/页面/API/数据库）
- 格式：`Emoji + 执行者名称`
- 常用前缀：`👤 用户` / `📄 [页面名]` / `⚙️ 后端 API` / `🗄️ 数据库`

## §4 功能点详细设计（核心章节）

这是 PRD 的主体。**每个功能点是自包含的编码单元**，固定 6 个子节结构——但每节内容"有无"由实际需求决定。

#### 示例：项目管理 — 项目列表功能点（F-M1-01）的业务规则

```markdown
#### 4.1.4 业务规则

##### 4.1.4.1 查询（List）

**查询规则**：

| # | 规则 | 说明 |
|---|------|------|
| B-M1-01 | 默认按「更新时间倒序」排列 | 最新修改的项目在最前 |
| B-M1-02 | 支持按项目名称（模糊匹配）、状态、负责人筛选 | 名称搜索使用后端分词，最小 2 字符触发 |
| B-M1-03 | 已归档项目不在默认列表中展示 | 需要切换到「包含已归档」视图才可见 |
| B-M1-04 | 用户只能看到自己是成员的项目 | 非成员项目即使知道 URL 也无法访问（接口层权限拦截）|

**分页**：默认 pageSize=20，可选 10/50/100，不支持无限滚动。

##### 4.1.4.2 新增（Create）

**前置校验**：

| 字段 | 校验规则 | 失败行为 |
|------|---------|---------|
| 项目名称 | 必填，1-50 字符，同一用户下不可重名 | 拦截并提示"项目名称已存在" |
| 负责人 | 必填，必须是系统内有效用户 | 拦截并提示"请选择有效负责人" |

**业务处理逻辑**：

| # | 步骤 | 规则 | 注意点 |
|---|------|------|--------|
| B-M1-05 | 重名检测 | 在当前用户的所有项目中（含已归档）检查名称唯一性 | 大小写不敏感，前后空格自动 trim 后比较 |
| B-M1-06 | 自动创建者加入 | 项目创建者自动成为项目成员（角色=管理员） | 即使创建者在「负责人」字段选了别人 |
| B-M1-07 | 初始状态 | 新建项目状态 = draft（草稿），不触发任何通知 | 只有首次「提交审核」才进入审批流 |
| B-M1-08 | 默认看板 | 自动创建 4 个默认列：待办 / 进行中 / 待验证 / 已完成 | 用户可在设置中自定义列，但初始化必须有这 4 个 |

##### 4.1.4.3 修改（Update）

**可编辑性矩阵**：

| 字段 | 可编辑 | 编辑条件 | 不可编辑原因 |
|------|:------:|---------|------------|
| 项目名称 | ✅ | draft / returned 状态下 | published 后改名会影响引用稳定性 |
| 负责人 | ✅ | 始终 | — |
| 描述 | ✅ | 始终 | — |
| 状态 | ❌ | — | 状态变更走独立的状态流转操作，不在编辑表单中 |
| 创建时间 | ❌ | — | 系统生成，不可修改 |

**业务处理逻辑**：

| # | 步骤 | 规则 | 注意点 |
|---|------|------|--------|
| B-M1-09 | 并发控制 | 使用乐观锁（version 字段），版本冲突时返回 409 + 提示刷新 | 前端需处理 409 响应，自动 reload 后重试 |
| B-M1-10 | 名称修改审计 | 项目名称变更记录到 operation_log 表（旧值→新值+操作人+时间）| 用于追溯和回滚判断 |
| B-M1-11 | 状态锁定 | published 状态下禁止修改名称和负责人 | 如需修改，必须先「撤回草稿」或走变更申请流程 |

##### 4.1.4.4 删除（Delete）

**删除方式**：逻辑删除（设置 `deleted_at` + `deleted_by`），不物理删除。

**前置条件检查**：

| 条件 | 检查内容 | 不满足时行为 |
|------|---------|-------------|
| 成员数 | 项目下不能有除创建者以外的活跃成员 | 提示"请先移转或移除其他成员" |
| 关联任务 | 不能有未完成（non-done）的任务 | 提示"该项目下有 N 个进行中的任务，请先完成或转移" |
| 状态限制 | 只有 draft / archived 状态可删除 | published / reviewing 状态下禁用删除按钮 |

**业务处理逻辑**：

| # | 步骤 | 规则 | 注意点 |
|---|------|------|--------|
| B-M1-12 | 二次确认 | 删除前弹窗显示"确定删除「XXX」？删除后可在回收站 30 天内恢复" | 弹窗必须显示项目名称，不能用通用文案 |
| B-M1-13 | 级联处理 | 子数据（任务/看板列/文档）标记为 `orphaned`（孤儿态）| 不级联删除子数据，保留数据完整性用于恢复 |
| B-M1-14 | 恢复机制 | 回收站保留 30 天，超期后由定时任务物理清理 | 恢复时同时恢复 orphaned 子数据的关联关系 |
```

**验收标准编写要求**：
