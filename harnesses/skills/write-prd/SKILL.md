# Write PRD

## Overview

结构化产品需求文档编写 Skill。核心原则：**先骨架后血肉**——先定业务流程和功能清单（§2+§3），再逐功能点填充详细设计（§4）。输出物必须同时满足人可读和 AI 可消费两个目标。

## When to Use

```
需要输出 PRD？
  ├── 是 → 产品原型阶段 / SDLC Step 2 → ✅ 使用本 Skill
  ├── 是 → 需求已明确但缺少结构化规格 → ✅ 使用本 Skill
  └── 否 → 不使用
```

**不适用场景**：
- 技术方案/架构设计（归 Step 3）
- 一句话需求描述（不需要完整 PRD）
- 纯 UI 原型（无行为逻辑）

## Core Pattern: Skeleton-First Writing

| 阶段 | 输出章节 | 关键动作 | 禁止事项 |
|------|---------|---------|---------|
| **1. 定位** | §1 概述 | 写清模块定位、用户角色、前置依赖 | 不要跳过，即使看起来简单 |
| **2. 骨架** | §2 业务流程 + §3 功能清单 | 先画 Happy Path 流程图 → 拆功能点 → 编号 F-Mx-NN | 不要从 §4 开始填 |
| **3. 血肉** | §4 功能点详细设计 | 逐个功能点按 6 子节模板填充 | 不要遗漏 .3 交互行为或 .5 数据规格 |
| **4. 收口** | §5 跨功能规则 + §6 验收标准 | 提取全局规则、写可测试的 AC | AC 不能是"功能正常" |

## Quick Reference

### 文档结构（6 章）

| 章 | 内容 | 必填 | 产出量级 |
|----|------|:----:|---------|
| §1 概述 | 定位 / 角色 / 依赖 | ✅ | ~30 行 |
| §2 业务流程 | Happy Path + 异常分支 + 页面交互 | ✅ 主流程必填 | ~60 行/Mermaid |
| §3 功能范围 | 清单(编号) / Out of Scope / 术语 | ✅ | ~20 行 |
| §4 功能点详细设计 | 每功能点 6 子节（见模板） | ✅ 核心 | ~80 行/功能点 |
| §5 跨功能规则 | 全局状态机 / 校验 / 交互约定 / 权限 | 按需 | ~40 行 |
| §6 验收标准 | 功能 AC / 边界异常 AC / UI/UX AC | ✅ | ~30 行 |

### 编号体系

| 前缀 | 格式 | 示例 |
|------|------|------|
| `F-` | 功能点 | `F-M1-03` |
| `B-` | 业务规则（功能点内） | `B-M1-05` |
| `G-` | 全局规则（跨功能点） | `G-M1-02` |
| `AC-` | 验收标准 | `AC-M1-12` |

### Mermaid 节点约定

| 含义 | 语法 | 示例 |
|------|------|------|
| 用户操作 | `[文本]` | `[点击新建按钮]` |
| 系统/页面 | `([文本])` | `(显示创建表单)` |
| 判断 | `{文本}` | `{校验通过？}` |
| 数据/状态 | `[[文本]]` | `[[status=draft]]` |
| API | `>文本]` | `>POST /api/tasks]` |

## Implementation

### 完整模板 → [prd-template.md](./prd-template.md)

6 章完整模板，含每个子节的填写示例和格式规范。编写时以此为准。

### 检查清单 → [prd-checklist.md](./prd-checklist.md)

完成后逐项检查：结构完整性、一致性校验（状态机/字段/功能点/验收覆盖）、质量门禁。

## Common Mistakes

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 从 §4 开始线性填充 | 功能点遗漏、编号跳跃 | 先 §2+§3 定骨架，确认功能点全了再填 §4 |
| 验收标准写"功能正常" | 无法测试、无法自动化 | 用 Given-When-Then 格式写可执行步骤 |
| 状态机与枚举值不一致 | 编码阶段发现矛盾 | 写完状态机立即做对照表校验 |
| AI 编码提示写成技术方案 | 与 Step 3 重复、膨胀 | 只记录实现陷阱和注意点 |
| 跳过 Out of Scope | 范围蔓延、评审时被挑战 | 明确写出"不做什么"及原因 |
| §5 与 §4 规则重复 | 维护时改了一处漏另一处 | 全局规则只放 §5，单功能规则放 §4.X.4 |

## Red Flags — STOP and Check

出现以下信号时停下来检查：

- §3 功能清单中有某项在 §4 找不到对应设计 → **补齐 §4 或从 §3 移除**
- §6 验收标准中存在"正常"、"正确"、"可用"等模糊词 → **重写为可执行步骤**
- 状态机的状态列表与数据规格枚举值数量不同 → **做对照表逐项核对**
- §4.X.5 数据规格中的字段不在 §4.X.1 领域模型中 → **补充领域模型或删除多余字段**
- 文档中出现 TBD / TODO / 待补充 → **要么现在填上，要么标记为明确的后续版本**

**以上任何一项未通过 = PRD 未完成。**
